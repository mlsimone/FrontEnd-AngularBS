In November 2021, I left my job at ECi Software Solutions to focus on my health and learning new software skills. The time away from the demands of a 40 hour week, gave me the opportunity to build this web site from the ground up, and showcase my capabilities as a software developer.

My vision for the site started out simple: build an e-commerce type web site where I can display and sell items I've collected. I know there are several e-commerce sites available that get the foot traffic needed to sell items on the internet. However, selling my collection, was really not what I was going for. I was going for something that could illustrate my capabilities:

Security, responsiveness, speed, and ease of use were all top priorities. I wanted the ability to add an item(s) to the inventory database, and upload as many images as I wanted.

After careful consideration of my requirements, I opted for a solution consisting of 2 distinct layers, giving a "separation of concerns":

  • A presentation layer, written in Angular, and hosted in an Azure Static Web App. This piece of code executes mostly in the browser.

  • A data layer, written in C#, to store and retrieve all the item details and images. This piece of code is also hosted in Azure and executes in an App Service which has access to an SQL database and Blob Storage.

The two layers communicate, by sending and receiving item data between each other based on browser input.

You can go anywhere in the site that you want. However, you will get a 401 (Unauthorized) if you attempt to add an item. Only specific users are authorized to add items.

The following are some bullet points about my site, that I'd like to share:

  • The Move Away from One Giant Monolithic Solution:

    I chose to divide the solution into two distinct software projects for a couple reasons. I wanted the project to use compute resources efficiently. With Angular, the web browser executes all the UI code, and validates input on forms, so minimal UI work is executed in Azure. Azure resources are mostly focused on processing data requests coming from the browser.

    The second reason is that dividing the project into 2 distinct pieces is a time-saver when it comes to testing. For example, if I make a change to the presentation layer (UI), I don't need to test the "data storage" logic, and other logic far from the UI change.

    Modern software development is moving away from Monoliths for a couple reasons: They don't easily allow teams of people to work together efficiently. With the presentation layer broken out into its own project, artistic software developers with a background in HTML and bootstrap can focus on writing the code which delivers a visually pleasing UI experience, far away from the other logic. This allows everyone to work harmoniously in parallel on what they do best. When logic is all mixed up in the various source files of a monolith, the chances of a developer having to "wait" until another developer checks a source file goes up.

    Last but not lease, the software is more organized, making it is easier to find logic. There is less customization and more standardization of where logic belongs, which is helpful when a new developer comes on board. When creating a new project in most frameworks, you will find folders for logic such as Controllers, Models, Views, and Data.

  • Github and Creating New Feature Releases:

    The presentation layer and the data layer each have their own github project/repository in my github account. I use the term application, repository or project, interchangeably. Keep in mind that they are equivalent to the term "layer" used above. For example, the presentation application, repository or project, and the data application, repository or project. When I was ready to add a new feature to either application, I checked out a new feature branch onto my development box, where I would write and test the software, per the requirements. Once I felt comfortable with the testing outcomes, the new feature release would get "git PUSHed" from my development box up to my github account. Lastly a pull request was executed to merged the new feature branch into the main branch.

  • Azure Deployment: via Github Actions Workflow

    Each repository in my github account, has its own Github Actions workflow. These workflows are prescribed by Microsoft as the only way to deploy software to their cloud. The workflow in the presentation layer compiles and deploys the Angular project to its Azure cloud (a Static Azure Web Site), while the workflow in the data layer compiles and deploys the C# Web API project to its Azure cloud (an App Service). Both workflows are triggered to deploy software to Azure on a "git PUSH" to the repositories. Note: if there are any compilation problems with the code pushed to github, the deployment will fail, and keep the existing version of software. I also added another trigger, workflow_dispatch that allowed me to force deployment to the Azure clouds from github, as I needed.

  • Testing and Troubleshooting Issues with the Applications in the Cloud:

    Logging was an absolute must for me when testing the data application in the Azure cloud. I found the eventlog to be the easiest log file to read. All the timestamps are in chronologic order, which was not the case with the logging stream you can view directly from the Azure portal. There is really no way to see where failures occur, unless you do a good job of anticipating where problems in your code can occur, and responding to those problems by logging them as warnings, errors, and exceptions.

  • Authentication: via Microsoft Identity Platform

    The login button in the Navigation Bar redirects the browser to a Microsoft Endpoint which executes 2 factor authentication to verify the identity of a given individual. Once Microsoft verifies a user's identity, it returns an ID token back to the browser where where the Navigation Bar is personalized to display "Hello, [individual's displayname]".

    Adding login and logout functionality into the presentation layer started with downloading the Angular Microsoft Authentication Library (MSAL) from npm, into my Angular project. I then created a new feature branch in github and made the appropriate software changes to configure MSAL within the presentation layer. Once these changes were "git pushed" from my development box to my github account, an Azure deployment was triggered, sending these software changes to the Azure cloud.

    In addition to the software changes made in the presentation layer, the presentation "application" also needed to be registered in Microsoft Entra ID .

  • Authorization and Security: via MSAL Interceptor

    MSAL not only captures your login details, but is also configured to give users access to protected resources, such as the item catalog, AFTER you login. The presentation layer was altered again to inject an access token into a given communication request going to the data layer. Once the data layer receives the communication request from the presentation layer, it will inspect the request to determine if an access token is present, and whether the access token has the appropriate SCOPE. The data layer will send the requested data back to the presentation layer, if and only if, the access token has the correct SCOPE. Without the access token, users will see an error message (unauthorized, 401).

    I allow free access to view and search thru the item catalog without a login. However, you need a successful login to add items to the catalog. In order to secure the item catalog, a new feature branch of the data layer was created. From there I added the appropriate changes to my controllers to secure them, such as [Authorize] and [RequiredScope]. Once these changes were "git pushed" from my local development box up to my github account, an Azure deployment was triggered, sending these software changes to the Azure cloud.

    In addition to the software changes made in both the presentation and data layers, the data "application" also needed to be registered in Microsoft Entra ID .

  • Managed Identities : Passwordless access to Protected Resources in Azure (eg. databases, storage)

    My data layer needs to have sufficient privileges to read and write to an SQL database and Blob Storage. In order for my application to access these protected resources in Azure, I created a Managed Identity, and modified my data layer to use the DefaultAzureCredential . The connection string in my application's configuration file contains a reference to a managed identity, rather than a password, making things more secure, as developers come and go. I found this article very helpful in understanding how Managed Identities work.

    The setup for using a managed identity is fairly involved. Persistence will get you to the finish line.

  • Site Responsiveness using Bootstrap:

    I wanted the site to run in a browser on any device, whether it be a phone, tablet, or desktop monitor. In order to support this, I downloaded the latest version of Bootstrap from npm and used the bootstrap classes within all my HTML elements so they respond appropriately as the size of the browser window changes. One thing you will notice as you expand or shrink the size of the browser, is that the HTML elements adjust to the current size of the browser. For example, the Navigation Bar turns into a hamburger icon if the browser is sufficiently small enough -- as would be the case in a phone's browser.

  • Azure Resources, such as hosting environments, databases and storage accounts:

    As a developer, I appreciate the ease and flexibility that Azure provides, without the headache of managing servers and patch updates. However, these benefits come at a cost, depending on the choices you make when you select servers in your Azure tenant, to host your site or run your database.

    There are options to use free resources, which I use when possible. For example, I use a "free SQL database" at the beginning of each month when my CPU time is replenished. However, once I exhaust all my free CPU time in a given month, I have it set up to charge me for "overages" so that I can keep my site up and running. Once you go beyond the "free time", costs can start to rack up with each passing day.

    Once I received my 1st Azure bill from Microsoft, I learned how to control my costs daily, by either shutting down resources or deleting them altogether if I no longer needed them. Sometimes, I "shut down the data layer", so I don't get charged for the database overages.
